# SquareGPSTest

Тестовое задание SquareGPS: offline-first мониторинг GPS-трекеров на iOS.

## Как запустить

1. Открыть `SquareGPSTest.xcodeproj` в Xcode.
2. Проставить свои данные в Signing & Capabilities таргета.
3. Сбилдить для симулятора или устройства.

Проект использует демо-учётные данные из задания:

- `login`: `demo-eu@navixy.com`
- `password`: `123456`
- API base URL: `https://api.eu.navixy.com/v2/`

## Что реализовано

### Core: offline-first список

Реализованы три экрана (продуктово – 4):

- список трекеров;
- детализация выбранного трекера;
- карта со всеми трекерами;
- карта с конкретным трекером:

Список загружает трекеры через `tracker/list`, отображает `label` и поддерживает pull-to-refresh. Детализация показывает поля `label`, `model`, `device_id`.

Offline-first поведение:

- трекеры сохраняются локально в SwiftData;
- при старте приложение сначала читает локальный кэш и показывает его в UI;
- refresh из сети обновляет локальную БД;
- если refresh завершается ошибкой, текущий список остаётся на экране, но показывается сообщение об offline/error состоянии.

Локальная БД остаётся источником сохранённых данных. Для UI поверх неё используется небольшой Combine bridge: репозиторий читает актуальные данные из SwiftData и публикует их через `CurrentValueSubject`. Такой подход оставляет offline-first контракт, но при этом позволяет мёржить разные источники данных внутри Repository, оповещать потребителей об изменении данных / снабжать их данными по подписке.

### Auth и сеть

Реализованы эндпоинты:

- `user/auth`;
- `tracker/list`;
- `tracker/get_states`.

Session hash сохраняется в Keychain (безопасное хранилище для сенсетив данных), чтобы не выполнять `user/auth` на каждый запуск. При ошибках авторизации нетворк сервис умеет инвалидировать hash, повторить запрос со свежей сессией и поставить в очередь все остальные запросы где нужен hash, пока ждём новый валидный hash.

Все HTTP-запросы и ответы логируются в консоль через network logger.

### Optional B: карта

- экран MapKit с трекерами на карте;
- подпись `tracker.label` рядом с маркером;
- polling `tracker/get_states` раз в 5 секунд;
- обновление координат на карте;
- поворот маркера по `heading`;
– режим "слежения" за трекером(ми);
- остановка polling при уходе с экрана и при уходе приложения из active state/background;
- error/loading состояние на карте;
- возможность открыть детали по тапу на трекер на карте.

## Архитектура

Проект разделён на несколько слоёв:

- `UI` — SwiftUI-экраны и ViewModel для списка, деталей и карты.
- `Routing` — навигация приложения и протоколы роутинга.
- `Repositories` — работа с данными: бд, походы в сеть и мёрж этих данных вместе.
- `Networking` — базовый URLSession-клиент, Navixy API-сервисы, DTO, domain-модели и маппинг.
- `Logging` — консольное логирование network/keychain операций.

Навигация вынесена в `AppRouter`. ViewModel не создают экраны напрямую: они зависят от узких routing-протоколов (`OnMapRouting`, `OnTrackerDetailsRouting`) и вызывают намерения вроде `openMap` / `openDetails`. Это держит UI-сценарии читаемыми и не смешивает пользовательские действия с построением navigation destination.

Naming conventions:

- `..Service` - интерфейсы/сущности для похода в конкретный эндпоинт / набор эндпоинтов;
- `..Repository` — сущности для работы с данными: внутри может быть поход в сеть, доступ к бд и мёрж этих данных вместе, а потребитель получает готовые данные без знания как именно они были получены. на данные репозитория могут подписываться разные потребители;
- `..DTO` — структуры ответа API;
- `..Domain` — сетевые доменные сущности после DTO-маппинга;
- `..Model` — UI-модели для связи View и ViewModel;
- `Stored...` — SwiftData-модели локального хранения.

## Trade-offs

- Отдельный экран логина не реализовывал: демо-логин и пароль захардкожены в composition root, как разрешено в задании.
- Optional A без карты пропущен осознанно, потому что Optional B покрывает тот же polling/get_states сценарий и дополнительно визуализирует координаты и heading.
- Автотесты не добавлял.
